🔒 Seguridad — Fase 1
JWT NUNCA en localStorage
Vulnerable a XSS. Access token solo en Zustand memory, refresh en HttpOnly cookie.
HTTPS obligatorio desde día 1
SECURE_SSL_REDIRECT=True en prod. HSTS 1 año. Igual que ferrelonstock settings_prod.py.
Rate limiting en endpoints auth
Máx 5 intentos login por IP/hora. 3 reenvíos OTP por hora. django-ratelimit.
Contraseñas con Argon2
Más seguro que bcrypt. django[argon2] en dependencies. PASSWORD_HASHERS en settings.
CORS estricto
Whitelist solo el dominio del frontend. Nunca CORS_ALLOW_ALL_ORIGINS=True en producción.
Variables de entorno separadas
SECRET_KEY, DB creds, API keys NUNCA en código. .env en .gitignore. django-environ.

Arquitectura escalable

3 tiers de crecimiento · Sin cambiar el stack core
Tier 1 · MVP
0 – 10k usuarios
Railway + Vercel + PostgreSQL gestionado. Un proceso Django. Redis shared. Cloudinary para media. Todo el stack actual sin cambios.
Railway free tierVercel hobby PostgreSQL managedRedis shared Cloudinary free~$0/mes
Tier 2 · Crecimiento
10k – 100k usuarios
Múltiples workers Django. Redis dedicado para cache + Celery. Read replica PostgreSQL para queries pesadas. CDN Cloudflare para assets. Índices DB optimizados.
Railway paidRedis dedicado PG read replicaCloudflare CDN DB indexes audit~$50-200/mes
Tier 3 · Escala
100k+ usuarios
Migración a Kubernetes. Django separado del Celery worker. Chat en servicio independiente. ElasticSearch para búsqueda. PostGIS dedicado. Message queue (RabbitMQ).
K8s / ECSElasticSearch Chat microserviceRabbitMQ PG horizontal$500+/mes
Decisiones de diseño para escalar desde día 1
• API versionada (/api/v1/) para cambios sin breaking changes
• Paginación cursor (no offset) — escala a millones de registros
• Cache Redis en queries frecuentes desde el MVP
• Celery para toda operación que tarde más de 200ms
• Cloudinary para media (nunca en el servidor)
• PostGIS desde el inicio (migrar después es doloroso)
• JWT stateless (sin sesiones en DB) — escala horizontal sin estado
Trampas de escalabilidad a evitar
• No usar select_related/prefetch_related → N+1 queries matan la DB
• No procesar emails/push en el request handler → timeouts
• No cargar imágenes originales sin redimensionar → bandwidth
• No hacer queries geoespaciales sin índice GIST en PostGIS
• No usar offset paginación en tablas grandes → lento
• No guardar sesiones en DB con muchos usuarios → cuello de botella
• No olvidar agregar indexes en FK y campos filtrados frecuentemente
